# 5교시. 지능형 AI 어시스턴트 만들기

## 1. 실습 개요

이번 실습에서는 IBM watsonx Orchestrate에서 여러 개의 AI 에이전트를 생성하고, 최종적으로 하나의 대표 에이전트가 사용자 요청을 받아 적절한 하위 에이전트에게 작업을 위임하는 **지능형 AI 어시스턴트**를 구현합니다.

실습 시나리오는 **창고 운영 관리자(Warehouse Manager)** 업무를 기반으로 합니다. 창고 관리자는 다음과 같은 일을 처리해야 합니다.

- 현재 창고 도크 상태 확인
- 잉여 재고가 발생한 트럭과 SKU 확인
- 잉여 재고 처리 방안 확인
- 관련 부서에 보낼 알림 이메일 작성

이번 시간의 핵심은 하나의 에이전트가 모든 일을 직접 처리하는 것이 아니라, 역할이 분리된 여러 에이전트를 만들고, 대표 에이전트가 요청에 맞는 하위 에이전트를 선택하도록 구성하는 것입니다.

---

## 2. 학습 목표

이 실습을 완료하면 다음 내용을 이해할 수 있습니다.

- watsonx Orchestrate에서 AI Agent를 생성하고 배포하는 방법
- Agent Description과 Behavior Instructions가 에이전트 동작에 미치는 영향
- 여러 하위 에이전트를 하나의 대표 에이전트에 연결하는 방법
- 사용자의 자연어 요청을 적절한 에이전트로 라우팅하는 방식
- 멀티 에이전트 기반 지능형 AI 어시스턴트의 기본 구조

---

## 3. 완성할 에이전트 구성

이번 실습에서는 총 4개의 에이전트를 만듭니다.

| 구분 | 에이전트 이름 예시 | 역할 | Home page 노출 |
|---|---|---|---|
| 하위 에이전트 1 | `Seongman_DockStatusAgent` | 창고 도크 상태 조회 | 해제 |
| 하위 에이전트 2 | `Seongman_SurplusAgent` | 잉여 재고 처리 방안 추천 | 해제 |
| 하위 에이전트 3 | `Seongman_SecretaryAgent` | 알림 이메일 초안 작성 | 해제 |
| 대표 에이전트 | `Seongman_AskWarehouseManager` | 사용자 요청을 적절한 하위 에이전트로 라우팅 | 체크 |

> 최종 사용자는 `AskWarehouseManager` 에이전트와만 대화합니다.  
> 나머지 에이전트들은 사용자가 직접 호출하지 않고, 대표 에이전트가 필요에 따라 호출합니다.

# 8. watsonx Orchestrate 콘솔 접속

1. IBM Cloud에 로그인합니다.
2. Resource list로 이동합니다.
3. `AI / Machine Learning` 영역에서 `watsonx Orchestrate` 리소스를 선택합니다.
4. `Launch watsonx Orchestrate` 버튼을 클릭합니다.
5. watsonx Orchestrate 콘솔이 열리는지 확인합니다.

콘솔을 처음 열면 첫 번째 에이전트를 생성하라는 팝업이 나타날 수 있습니다. 이 경우 `Skip for now`를 클릭합니다.

# 9. Dock Status Agent 생성

## 9.1 에이전트 생성

1. watsonx Orchestrate 콘솔에서 `Create or Deploy` 또는 `Create new agent`를 클릭합니다.
2. `Create agent`를 선택합니다.
3. `Create from scratch`를 선택합니다.
4. 다음 정보를 입력합니다.

### Name

```text
<본인이름>_DockStatusAgent
```

예시:

```text
Seongman_DockStatusAgent
```

### Description

아래 문장에서 `<생성한 에이전트명>`을 본인이 만든 에이전트명으로 변경합니다.

```text
<생성한 에이전트명>은 현재 창고 도크 상태에 대한 문의에 답변하는 데 특화되어 있습니다. 이 에이전트는 각 도크에서 어떤 트럭이 상차 및 하차 중인지, 해당 트럭이 어떤 제품을 운반하는지에 대한 상세하고 최신의 정보를 보유하고 있으며, 이 정보를 사용자에게 자세한 텍스트 형태로 제공합니다.
```

5. `Create`를 클릭합니다.

# 11. Secretary Agent 생성

## 11.1 에이전트 역할 이해

Secretary Agent는 잉여 재고와 관련된 알림 이메일 초안을 작성하는 역할을 합니다.

이 에이전트는 실제 이메일을 발송하지 않습니다. 사용자가 요청한 조건에 따라 이메일 제목과 본문을 생성하는 역할만 수행합니다.

---

## 11.2 에이전트 생성

`Manage agents` 화면에서 `Create agent`를 클릭합니다.

### Name

```text
<본인이름>_SecretaryAgent
```

예시:

```text
Seongman_SecretaryAgent
```

### Description

```text
<생성한 에이전트명>은 창고 관련 주제의 이메일을 작성하는 데 특화되어 있습니다.
```

입력 후 `Create`를 클릭합니다.

---

## 11.3 모델 및 Agent style 설정

- Model: `llama-3-405b-instruct` 또는 강사가 지정한 모델
- Agent style: `Default`

---

## 11.4 Behavior Instructions 입력

`Behavior` 섹션의 `Instructions`에 아래 내용을 입력합니다.

```text
페르소나:
- 당신은 이메일 초안을 작성하는 비서 역할입니다. 사용자가 특정 주제에 대한 이메일 작성을 요청하면, 해당 이메일의 텍스트 초안을 반환해야 합니다.

컨텍스트:
- 재고 내 잉여 재고와 관련된 내용을 주제로, 간결하고 전문적인 이메일 초안을 작성하세요.
- 이메일은 어떠한 서두 설명도 없이 바로 제목 줄로 시작하고, 그 다음 본문이 이어지도록 작성하세요.
- 이메일 작성에 대한 일반적인 지식을 바탕으로 구조와 어조를 구성하되, 특정 팀이나 미리 정의된 예시에만 제한되지 않도록 하세요.
- 별도의 지정이 없는 경우, 수신자와 내용은 일반적인 상황으로 가정하세요.
- 명시적으로 요구되지 않는 한, 지식의 한계나 특정 팀 언급은 피하세요.
- 아래에는 사용자 입력 예시와 그에 따른 생성 이메일 예시가 제공됩니다. 이를 참고하여 유사한 형식으로 작성하세요.
```

# 12. Warehouse Manager Agent 생성

## 12.1 대표 에이전트 역할 이해

Warehouse Manager Agent는 사용자가 직접 대화하는 대표 에이전트입니다.

이 에이전트는 다음 역할을 수행합니다.

- 사용자의 요청을 해석합니다.
- 요청 내용이 도크 상태 조회인지, 잉여 재고 처리인지, 이메일 작성인지 판단합니다.
- 적절한 하위 에이전트에게 작업을 위임합니다.
- 필요한 경우 여러 하위 에이전트를 조합하여 답변합니다.

---

## 12.2 에이전트 생성

`Manage agents` 화면에서 `Create agent`를 클릭합니다.

### Name

```text
<본인이름>_AskWarehouseManager
```

예시:

```text
Seongman_AskWarehouseManager
```

### Description

```text
<생성한 에이전트명>은 사용자 요청을 자신에게 연결된 하위 에이전트 중 가장 적절한 에이전트로 라우팅하는 역할을 담당합니다.
```

입력 후 `Create`를 클릭합니다.

---

## 12.3 모델 및 Agent style 설정

- Model: `llama-3-405b-instruct` 또는 강사가 지정한 모델
- Agent style: `ReAct`

---

## 12.4 하위 에이전트 연결

Warehouse Manager Agent 설정 화면에서 `Toolset` 영역으로 이동합니다.

1. `Agents` 섹션을 찾습니다.
2. `Add agent`를 클릭합니다.
3. `Add from local instance`를 선택합니다.
4. 앞에서 생성한 아래 3개 에이전트를 선택합니다.

```text
<본인이름>_DockStatusAgent
<본인이름>_SurplusAgent
<본인이름>_SecretaryAgent
```

5. `Add to agent`를 클릭합니다.
6. Toolset에 3개 하위 에이전트가 추가되었는지 확인합니다.

---

## 12.5 Behavior Instructions 입력

`Behavior` 섹션의 `Instructions`에 아래 내용을 입력합니다.

아래의 `<본인이름>`은 본인이 사용한 이름 또는 고유 식별자로 변경합니다.

```text
추론:
- 도크 상태와 관련된 작업에는 <본인이름>_DockStatusAgent 를 사용하세요.
- 잉여 재고와 관련된 작업에는 <본인이름>_SurplusAgent 를 사용하세요.
- 이메일 작성 작업에는 <본인이름>_SecretaryAgent 를 사용하세요.
- 사용자의 요청이 여러 작업을 포함하는 경우, 필요한 하위 에이전트를 순서대로 사용하여 종합 답변을 작성하세요.
- 사용자가 특정 트럭 ID, 도크 ID, SKU, 팀명을 명시하지 않은 경우, 가능한 범위에서 답변하되 필요한 정보가 부족하면 사용자에게 확인 질문을 하세요.
- 최종 답변은 사용자가 이해하기 쉽도록 요약과 세부 내용을 구분하여 작성하세요.
```

예시로 이름이 `Seongman`이라면 다음과 같이 입력합니다.

```text
추론:
- 도크 상태와 관련된 작업에는 Seongman_DockStatusAgent 를 사용하세요.
- 잉여 재고와 관련된 작업에는 Seongman_SurplusAgent 를 사용하세요.
- 이메일 작성 작업에는 Seongman_SecretaryAgent 를 사용하세요.
- 사용자의 요청이 여러 작업을 포함하는 경우, 필요한 하위 에이전트를 순서대로 사용하여 종합 답변을 작성하세요.
- 사용자가 특정 트럭 ID, 도크 ID, SKU, 팀명을 명시하지 않은 경우, 가능한 범위에서 답변하되 필요한 정보가 부족하면 사용자에게 확인 질문을 하세요.
- 최종 답변은 사용자가 이해하기 쉽도록 요약과 세부 내용을 구분하여 작성하세요.
```

# 15. 자주 발생하는 문제와 해결 방법

## 15.1 에이전트가 메인 채팅에 보이지 않음

### 원인

Warehouse Manager Agent의 `Home page` 체크박스가 해제되어 있을 수 있습니다.

### 해결 방법

Warehouse Manager Agent 설정 화면으로 이동하여 `Home page` 체크박스를 체크한 뒤 다시 배포합니다.

---

## 15.2 하위 에이전트가 메인 채팅에 보임

### 원인

Dock Status Agent, Surplus Agent, Secretary Agent의 `Home page` 체크박스가 체크되어 있을 수 있습니다.

### 해결 방법

각 하위 에이전트 설정 화면에서 `Home page` 체크박스를 해제한 뒤 다시 배포합니다.

---

## 15.3 대표 에이전트가 엉뚱한 답변을 함

### 원인

Warehouse Manager Agent의 Instructions에 하위 에이전트 이름이 정확히 입력되지 않았을 수 있습니다.

### 해결 방법

다음 항목을 확인합니다.

- 에이전트 이름에 오타가 없는지 확인합니다.
- 본인 이름 또는 고유 식별자가 실제 생성한 이름과 일치하는지 확인합니다.
- Toolset에 하위 에이전트 3개가 모두 추가되어 있는지 확인합니다.
- 수정 후 다시 배포합니다.

---

## 15.4 하위 에이전트가 호출되지 않는 것 같음

### 원인

하위 에이전트가 아직 배포되지 않았거나, Warehouse Manager Agent의 Toolset에 연결되지 않았을 수 있습니다.

### 해결 방법

1. `Manage agents`에서 하위 에이전트 3개가 모두 `Live` 상태인지 확인합니다.
2. Warehouse Manager Agent 설정 화면에서 Toolset의 Agents 목록을 확인합니다.
3. 누락된 에이전트가 있으면 `Add from local instance`로 추가합니다.
4. Warehouse Manager Agent를 다시 배포합니다.

---

## 15.5 답변이 매번 조금씩 다름

### 원인

LLM 기반 에이전트는 동일한 질문에도 표현이 조금씩 달라질 수 있습니다.

### 해결 방법

출력 형식을 더 고정하고 싶다면 Instructions에 다음과 같은 지시를 추가할 수 있습니다.

```text
응답은 반드시 다음 형식으로 작성하세요.
1. 요약
2. 상세 내용
3. 후속 조치
```

# 16. 실습 정리

이번 실습에서는 창고 운영 관리 시나리오를 기반으로 멀티 에이전트 구조의 지능형 AI 어시스턴트를 구현했습니다.

우리는 먼저 개별 업무를 담당하는 3개의 하위 에이전트를 만들었습니다.

- Dock Status Agent: 창고 도크 상태 조회
- Surplus Agent: 잉여 재고 처리 방안 추천
- Secretary Agent: 알림 이메일 초안 작성

그 다음, 이 하위 에이전트들을 Warehouse Manager Agent에 연결하여 하나의 대표 에이전트가 여러 업무를 조율하도록 구성했습니다.

이 구조는 실제 업무 시스템에서도 자주 활용될 수 있습니다. 예를 들어 HR, 마케팅, 영업, 고객지원, IT 운영 업무에서도 각 기능별 에이전트를 만들고, 대표 에이전트가 사용자 요청을 라우팅하는 방식으로 확장할 수 있습니다.

- 메인 채팅에서는 대표 에이전트만 사용한다.
- 도크 상태 조회, 잉여 재고 처리 조회, 이메일 작성 요청이 모두 정상 동작한다.
- 복합 요청에서 대표 에이전트가 여러 하위 에이전트를 조합하여 응답한다.
